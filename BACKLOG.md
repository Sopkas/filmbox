# Бэклог KinoPulse

Этот бэклог подготовлен для текущей кодовой базы (`frontend` + `backend`) и готов к разбиению на задачи в Linear.

## Обозначения приоритетов
- P0: Критично для стабильности, готовности к релизу и скорости команды.
- P1: Важные продуктовые и качественные улучшения после P0.
- P2: Желательные улучшения и задачи на масштабирование.

## P0: Стабилизация и предсказуемая поставка

### KP-001: Добавить CI-пайплайн для frontend и backend
- Приоритет: P0
- Объём: Добавить workflow GitHub Actions для установки зависимостей, сборки frontend и синтаксических проверок backend.
- Критерии приёмки:
  - CI запускается на каждый PR и push в main.
  - Сборка frontend (`npm run build`) обязательна.
  - Проверки backend через `node --check` для ключевых файлов обязательны.

### KP-002: Добавить базу линтинга и форматирования
- Приоритет: P0
- Объём: Подключить ESLint + Prettier для frontend и backend, добавить скрипты в оба `package.json`.
- Критерии приёмки:
  - В обоих приложениях есть `npm run lint`.
  - Кодовая база проходит линтинг без ошибок.
  - Команда форматирования и общая конфигурация задокументированы.

### KP-003: Ввести слой валидации API-запросов
- Приоритет: P0
- Объём: Валидировать body и query-параметры для auth, reviews, collections и movies.
- Критерии приёмки:
  - Некорректные payload отклоняются с единообразными ответами 400.
  - Ошибки валидации отдаются в одном формате.
  - Существующее корректное поведение не ломается.

### KP-004: Добавить интеграционные API-тесты для ключевых сценариев
- Приоритет: P0
- Объём: Покрыть signup/login, поиск, создание/обновление review, обновление top10, watch later, abandoned и collections.
- Критерии приёмки:
  - Тесты запускаются одной командой.
  - Для критичных endpoint есть позитивные и негативные кейсы.
  - Тесты выполняются в CI.

### KP-005: Устранить рассинхронизацию схемы для типа rating
- Приоритет: P0
- Объём: Перенести миграцию типа `rating` из runtime-логики роутов в детерминированный путь миграции/инициализации БД.
- Критерии приёмки:
  - Финальный тип и constraint для `rating` определены в схеме один раз.
  - При старте приложения больше не требуется runtime-проверка миграции столбца `rating`.
  - Совместимость существующих данных сохранена.

### KP-006: Добавить централизованное логирование и request ID
- Приоритет: P0
- Объём: Добавить middleware с correlation/request id и структурированные логи для ошибок API и ключевых действий.
- Критерии приёмки:
  - Каждый request-лог содержит request id, method, path, status и duration.
  - Логи ошибок содержат request id и контекст маршрута.
  - Чувствительные данные (пароли, токены) не логируются.

### KP-007: Усилить auth и security-настройки по умолчанию
- Приоритет: P0
- Объём: Добавить Helmet, безопасную стратегию CORS по окружениям и защиту auth-эндпоинтов от брутфорса/частых запросов.
- Критерии приёмки:
  - Security middleware включён в production-профиле.
  - На auth-эндпоинтах включён rate limit.
  - Security-настройки задокументированы в README.

### KP-008: Улучшить UX обработки ошибок на frontend
- Приоритет: P0
- Объём: Заменить тихие/console-only ошибки на видимые состояния ошибок и действия повтора в Tracker, Library и Collections.
- Критерии приёмки:
  - Ошибки API показываются пользователю с понятным текстом.
  - Состояния loading/empty/error чётко разделены на всех основных страницах.
  - Критичные действия не падают молча.

### KP-009: Исправить консистентность текстовой кодировки
- Приоритет: P0
- Объём: Нормализовать исходники в UTF-8 и убрать «битые» строки UI на некоторых страницах.
- Критерии приёмки:
  - Тексты UI корректно отображаются на всех страницах.
  - В исходниках нет mojibake-литералов.
  - Pre-commit/CI-проверка предотвращает регрессии кодировки.

### KP-010: Стандартизировать локальный bootstrap разработки
- Приоритет: P0
- Объём: Добавить одну корневую команду для запуска DB + backend + frontend и чётко задокументировать setup окружения.
- Критерии приёмки:
  - Полный локальный стек запускается одной документированной командой.
  - В корневом README перечислены все обязательные env-переменные.
  - Новый разработчик может запустить проект с нуля менее чем за 15 минут.

## P1: Продуктовые улучшения и UX

### KP-011: Добавить модалку/страницу деталей фильма
- Приоритет: P1
- Объём: Открывать подробную карточку фильма из результатов поиска и списков с постером, метаданными и пользовательскими действиями.
- Критерии приёмки:
  - Детали можно открыть минимум из двух точек входа.
  - В деталях отображаются данные источника и пользовательский контекст review.
  - Навигация и возврат работают корректно.

### KP-012: Добавить пагинацию/подгрузку для больших библиотек
- Приоритет: P1
- Объём: Уйти от загрузки всех фильмов профиля разом; добавить серверную пагинацию и фронтовую подгрузку страниц.
- Критерии приёмки:
  - API поддерживает page/limit или cursor.
  - На frontend можно загружать дополнительные страницы.
  - UI остаётся отзывчивым при 1000+ записях.

### KP-013: Добавить редактирование и удаление review
- Приоритет: P1
- Объём: Дать пользователю возможность редактировать и удалять отзывы из контекста tracker и library.
- Критерии приёмки:
  - Редактирование сохраняет валидацию и корректно обновляет timestamps.
  - Удаление удаляет review и безопасно обновляет связанный top10.
  - Для destructive-действий есть подтверждение в UI.

### KP-014: Добавить переименование и сортировку элементов в подборках
- Приоритет: P1
- Объём: Разрешить переименовывать подборки и менять порядок фильмов внутри подборки.
- Критерии приёмки:
  - Название подборки можно изменить inline или через модалку.
  - Порядок фильмов можно изменить и сохранить.
  - Превью подборки отражает сохранённый порядок.

### KP-015: Добавить кэш результатов поиска и ограничение частоты запросов
- Приоритет: P1
- Объём: Кэшировать недавние ответы поиска и сократить повторные обращения к внешнему API.
- Критерии приёмки:
  - Повтор одного и того же запроса в пределах TTL обслуживается из кэша.
  - Throttling/rate-limit защищает расход внешнего API.
  - Поведение кэша можно отследить по логам.

### KP-016: Провести accessibility-pass по основным страницам
- Приоритет: P1
- Объём: Улучшить клавиатурную навигацию, фокус-состояния, семантические лейблы и контраст.
- Критерии приёмки:
  - Ключевые действия доступны с клавиатуры.
  - Поля и кнопки имеют корректные доступные подписи.
  - Нет критичных accessibility-блокеров по автоматизированному аудиту.

### KP-017: Улучшить мобильный UX для tracker-страницы
- Приоритет: P1
- Объём: Доработать layout и взаимодействия для панелей, списков и форм tracker на узких экранах.
- Критерии приёмки:
  - Ключевые действия tracker доступны на ширине 360px.
  - Нет обрезанных контролов и наложений в основных сценариях.
  - Touch-элементы соответствуют минимальным требованиям по размеру.

## P2: Масштабирование, наблюдаемость и эксплуатация

### KP-018: Добавить Docker-профиль для всего стека в dev
- Приоритет: P2
- Объём: Добавить профиль Docker Compose, запускающий не только PostgreSQL, но и backend/frontend в контейнерах.
- Критерии приёмки:
  - Весь стек запускается одной compose-командой.
  - Dev-процесс с hot reload остаётся практичным.
  - В README есть troubleshooting для контейнерного запуска.

### KP-019: Добавить шаблоны production-деплоя
- Приоритет: P2
- Объём: Подготовить примеры/манифесты деплоя для backend и frontend, включая управление env-переменными.
- Критерии приёмки:
  - Есть минимум один документированный target деплоя (например Render/Fly/VM).
  - Задокументированы health-check и стратегия перезапуска.
  - Список обязательных секретов явно описан.

### KP-020: Добавить метрики и мониторинг доступности
- Приоритет: P2
- Объём: Экспортировать операционные метрики и настроить базовые алерты по доступности/ошибкам.
- Критерии приёмки:
  - Backend отдаёт health + ключевые метрики.
  - Настроены алерты на downtime и повышенный 5xx.
  - Ссылка на дашборд задокументирована.

### KP-021: Добавить playbook по backup/restore БД
- Приоритет: P2
- Объём: Определить расписание бэкапов и проверить инструкции восстановления.
- Критерии приёмки:
  - Команда/джоба бэкапа задокументирована и автоматизирована.
  - Восстановление протестировано на чистой БД.
  - Чеклист восстановления есть в документации репозитория.

### KP-022: Добавить аудит критичных пользовательских действий
- Приоритет: P2
- Объём: Логировать критичные действия: auth-события, изменения review, изменения подборок.
- Критерии приёмки:
  - Для критичных категорий действий пишутся audit-записи.
  - Определена политика хранения audit-данных.
  - Доступ к audit-данным ограничен.

## Рекомендуемая настройка Linear
- Создать labels: `backend`, `frontend`, `db`, `devops`, `security`, `ux`.
- Создать milestones: `MVP Stabilization`, `UX Improvements`, `Production Readiness`.
- Начать с P0, затем пересмотреть приоритеты P1/P2 после первого стабильного релиза.
